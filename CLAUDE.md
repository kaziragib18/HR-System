- **Fixed a real login-crashing bug, plus a systemic reliability gap it exposed**: login started failing for everyone with "Invalid email or password" even for correct credentials. Root cause was `createSession` (`apps/api/src/modules/auth/auth.service.ts`) ŌĆö it `create`s a `Session` row with a fixed literal placeholder (`refreshToken: 'pending'`, since the real signed token embeds the session's own id and can't be known before the row exists), then `update`s it with the real token. `refreshToken` is `@unique`. A single interrupted attempt (this environment's Supabase pooler drops connections often ŌĆö see `## Environment variables` below) left one row stuck at `refreshToken: 'pending'`, and every subsequent successful login's `create` call then collided with that same literal string, threw `P2002`, and crashed **the entire Node process** ŌĆö not just that one request. Fixed `createSession` to wrap both statements in one `prisma.$transaction` (a crash mid-way now rolls back instead of leaving a stale row) plus a random suffix as defense in depth. The process-crashing part was itself a symptom of a wider gap: every controller's `handle(res, err)` fell back to `throw err` for any error it didn't specifically recognize ŌĆö but these are async Express 4 route handlers with no `next(err)` wiring, so the re-thrown error became an **unhandled promise rejection**, which crashes the whole process by default on Node 15+. This exact `throw err` pattern was duplicated across all 11 controllers (auth, leave, attendance, employees, payroll, salary, departments, documents, announcements ŌĆö grep `throw err$` in `apps/api/src/modules/*/*.controller.ts` if it reappears). Replaced all 11 with a new `sendUnexpectedError(res, err)` in `apps/api/src/utils/response.ts` ŌĆö logs the error and always sends a real `500` response instead of letting it escape the request. Added `process.on('unhandledRejection'|'uncaughtException', ...)` in `index.ts` as a last-resort net for anything outside a controller's own try/catch (logs via the shared `logger` instead of an unattributed crash; `uncaughtException` still exits after logging, since process state may be corrupted, but nodemon/PM2 restart it cleanly). Separately, `LoginForm.tsx` was masking *any* failure ŌĆö network-down, a real 500, or an actually-wrong password ŌĆö into the same fixed "Invalid email or password" string, since it only ever read `err.response?.data?.error` and silently fell back otherwise; now distinguishes a real server response (shows its actual message) from no response at all (shows "Cannot reach the server..." instead of blaming the password).

## Architecture

### Monorepo layout

```
apps/api/      ŌåÆ Express backend (port 4000)
apps/web/      ŌåÆ Next.js 14 App Router frontend (port 3000)
packages/types ŌåÆ Shared TypeScript enums + interfaces (zero deps)
packages/utils ŌåÆ Pure functions: date, leave calc, tax engine, currency, pagination
prisma/        ŌåÆ Single schema.prisma + migrations + seed scripts
```

`packages/types` has no runtime dependencies ŌĆö everything else imports from it. Always add new enums and shared interfaces there, never redefine them locally.

### Multi-tenancy: BD vs UK

The `Office` model is the tenancy anchor. Every resource (Employee, LeaveType, PayrollRun, PublicHoliday, etc.) carries an `officeId`. `office.middleware.ts` attaches `req.officeId` from the authenticated user's employee record; `SUPER_ADMIN` bypasses this and can query any office.

Office-specific logic (tax, currency, timezone, leave entitlements) is resolved by `officeCode` in `packages/utils` ŌĆö never inline BD/UK conditionals anywhere else.

| | Bangladesh (BD) | United Kingdom (UK) |
|---|---|---|
| Currency | BDT | GBP |
| Timezone | Asia/Dhaka (UTC+6) | Europe/London (BST/GMT) |
| Tax | BD Income Tax slabs (FY 2024-25) | UK PAYE + National Insurance |

### Auth flow

Three JWT token types, all signed with keys from `apps/api/src/utils/jwt.ts`:

| Token | TTL | Secret | Purpose |
|---|---|---|---|
| Access token | 15 min | `JWT_SECRET` | Bearer auth on all API requests |
| Refresh token | 7 days | `JWT_REFRESH_SECRET` | httpOnly cookie ŌåÆ `/auth/refresh` |
| Temp token | 5 min | `JWT_SECRET` | Bridges login ŌåÆ 2FA (TOTP) verify step |

Access tokens are stored **in Zustand memory only** (`apps/web/src/store/auth.store.ts`) ŌĆö never in localStorage. The Axios client in `apps/web/src/lib/api/client.ts` handles silent refresh on 401 with a request queue to prevent parallel refresh races.

Every refresh token maps 1:1 to a `Session` row (device info, IP, `isValid`, `expiresAt`, `lastUsedAt`). `POST /auth/login` creates a session directly unless the user has `isTwoFactorEnabled`, in which case it returns a temp token instead and the session is only created after `POST /auth/2fa/verify` succeeds. Users manage their own active sessions via `GET /auth/sessions` / `DELETE /auth/sessions/:id` (`auth.service.ts`); TOTP secrets and enable/disable live behind `/auth/2fa/setup|enable|disable`, all `authenticate`-gated (not part of the login bridge).

### RBAC

`requireRole(...roles)` in `apps/api/src/middleware/rbac.middleware.ts` uses a numeric hierarchy:

```
SUPER_ADMIN(5) > HR_MANAGER(4) > DEPT_HEAD(3) > DEPT_MANAGER(2) > EMPLOYEE(1)
```

Passing a role allows that role **and all roles above it** ŌĆö this is why `HR_MANAGER` still passes every `requireRole(DEPT_MANAGER)` gate used on GET/list endpoints (office-wide **view** access to approvals is intentional). Use `requireExactRole` when a specific allow-list (not "this role or higher") is needed ŌĆö this is how `HR_MANAGER` is excluded from actually approving/rejecting: every leave/attendance-adjustment/late-excuse **write** endpoint uses `requireExactRole(DEPT_MANAGER, DEPT_HEAD, SUPER_ADMIN)` instead of `requireRole`, so HR_MANAGER can see the queue but has no action buttons. Use `selfOrRole(paramName, ...roles)` for routes where an employee should be able to act on their own record even without the listed role (e.g. uploading their own avatar or editing their own profile sub-resources) ŌĆö it passes if `req.params[paramName]` matches the caller's `employeeId` OR they hold one of `roles`+. `isSelfOrRole(user, targetEmployeeId, ...roles)` is the same check as a plain predicate, for use inside controllers/services when the target id isn't a route param (e.g. resolved from a DB row or a multipart body).

`DEPT_HEAD` is unique per department, enforced at write-time: `updateEmployeeRole` (`employees.service.ts`) rejects promoting a second person to `DEPT_HEAD` in a department that already has one (reassign/demote the existing head first). `Department.managerId` is always that department's `DEPT_HEAD`.

**Team-request approval routing** (leave, late-excuse review, attendance-adjustment review) is resolved via the real reporting chain, not a flat "any user with this role in the office" lookup ŌĆö see `apps/api/src/services/approver-resolution.service.ts`:
- An `EMPLOYEE`/`DEPT_MANAGER` requester's application routes to their `Employee.reportingToId` (their real `DEPT_MANAGER`) if set, else falls back to their department's `DEPT_HEAD` (`Department.managerId`).
- A `DEPT_HEAD` or `HR_MANAGER` requester's own application escalates to the `DEPT_HEAD` of the `HR` department specifically (`resolveHrDeptHeadApprover`) ŌĆö not to `SUPER_ADMIN`, and not to any other department's head. **Exception:** a `DEPT_HEAD`'s own *attendance-adjustment* request (not leave, not late-excuse) resolves to `HR_MANAGER` instead ŌĆö see below.
- Whoever resolves as approver can act on it; **additionally**, the requester's own department `DEPT_HEAD` can always act on any pending team request in their department even if it routed to a different `DEPT_MANAGER` (`canActOnTeamRequest`'s override) ŌĆö `SUPER_ADMIN` can always act on anything.
- `HR_MANAGER` never resolves as an approver and never passes `canActOnTeamRequest` ŌĆö by design, per the RBAC paragraph above, they are read-only on this whole flow, **except** for a `DEPT_HEAD`'s own attendance-adjustment request specifically (enforced inline in `attendance.service.ts`'s `reviewAdjustment`, not in `canActOnTeamRequest` itself ŌĆö see "Shipped after Milestones 1-4" for why this stayed a local carve-out rather than a shared-service change). No other `DEPT_HEAD`, including the HR department's own head, can act on it ŌĆö only `HR_MANAGER`/`SUPER_ADMIN`.

### API modules

Located in `apps/api/src/modules/`. Each module follows:
`<name>.routes.ts` ŌåÆ `<name>.controller.ts` ŌåÆ `<name>.service.ts` + `<name>.schemas.ts` (Zod)

Register the router in `apps/api/src/app.ts` under `/api/v1/<name>`.

Current modules: `announcements`, `approvals`, `auth`, `attendance`, `company`, `dashboard`, `departments`, `documents`, `employees`, `holidays`, `job-grades`, `leave`, `notifications`, `payroll`, `salary`

`approvals` is read-only: it aggregates cross-module history (leave approvals, attendance late-excuse review) into one unified, sorted timeline for the `/approvals` page ŌĆö it owns no Prisma models of its own. `company` manages office profile fields (name/address/logo) and compliance document uploads, both Super-Admin-gated.

`employees` is split across three controller/service pairs sharing one router (`employees.routes.ts`): `employees.*` (core CRUD, directory, org chart, bank info), `profile.*` (work experience, education, skills, certifications, identification ŌĆö each a small sub-resource CRUD keyed by `:id/<resource>/:subId`), and `avatar.controller.ts` (single-file avatar upload). Certification/identification uploads call `documents.service.ts`'s `uploadDocument` directly in-process (not over HTTP) so those files also show up as `Document` rows.

All responses must go through the helpers in `apps/api/src/utils/response.ts` (`sendSuccess`, `sendCreated`, `sendError`, `sendNotFound`, `sendForbidden`). Standard envelope: `{ success: bool, data?, error?, message?, meta? }`.

Middleware ordering in every protected route: `authenticate ŌåÆ officeScope ŌåÆ [requireRole] ŌåÆ [validate] ŌåÆ controller`

Cross-module logic that doesn't belong to a single module lives in `apps/api/src/services/`, not inside a module: `storage.service.ts` (Supabase signed URLs, used by `documents`), `notification.service.ts` (used by `attendance`, `leave`, `payroll` to write `Notification` rows), `twofa.service.ts` (used by `auth`), `leave-scheduler.service.ts` (the periodic leave-lifecycle sweep ŌĆö see "Scheduled jobs" below).

- **API client**: `apps/web/src/lib/api/client.ts` ŌĆö Axios instance with auth interceptors
- **React Query hooks**: `apps/web/src/lib/api/hooks/` ŌĆö one file per domain (e.g. `useEmployees.ts`, `useLeave.ts`)

### Real-time notifications

Notifications are written to the `Notification` table via Prisma. Supabase Realtime broadcasts the INSERT to the frontend ŌĆö no Socket.io or Redis ŌĆö via `apps/web/src/components/notifications/NotificationRealtimeProvider.tsx` (mounted globally inside the dashboard layout's `AuthGuard`), which subscribes through the Supabase browser client (`apps/web/src/lib/supabase.ts`) to `postgres_changes` INSERTs filtered to the current employeeId, feeds `apps/web/src/store/notification.store.ts` (bell badge), and also invalidates the `['notifications']` React Query cache so the `/notifications` list picks up the new row without needing to remount. Requires Realtime enabled on the `Notification` table in the Supabase dashboard ŌĆö this app authenticates via its own custom JWT rather than Supabase Auth, so the browser client only ever holds the anon key; if RLS is enabled on `Notification` with no permissive anon `SELECT` policy, Realtime events won't reach the client at all (falls back silently to `useUnreadCount`'s 60s polling for the badge; the list itself just won't update until the page is revisited). Neither of those Supabase-dashboard settings can be verified or fixed from this repo ŌĆö check them directly if realtime push isn't arriving for any user. **Security note**: the frontend's `employeeId=eq.` filter (`NotificationRealtimeProvider.tsx`) is a client-side subscription filter, not a security boundary ŌĆö no migration in this repo enables RLS or defines a policy on `Notification`. If the Supabase-dashboard side was never configured with an RLS policy restricting `SELECT` to the row's own `employeeId`, any authenticated client holding the anon key could subscribe without a filter and receive every employee's notifications over Realtime, not just their own. This needs to be confirmed directly in the Supabase dashboard (this app doesn't use Supabase Auth, so there's no `auth.uid()` to key a standard policy off ŌĆö it would need a custom claim or a service-mediated broadcast instead); it cannot be verified or fixed from this repo alone.

`createNotification` (`apps/api/src/services/notification.service.ts`) is fire-and-forget: failures are logged and swallowed, never surfaced to the caller, so a broken notification insert never blocks the underlying action (leave application, approval, etc.) from succeeding. Every team-request notification targets the single resolved reporting-chain approver/applicant (`resolveTeamApprover` ŌĆö see RBAC above) as its primary recipient, but new leave/late-excuse/attendance-adjustment *requests* (and leave cancellation requests) additionally broadcast to every `SUPER_ADMIN`/`HR_MANAGER` in the office via `notifyOfficeAdmins` ŌĆö see "Shipped after Milestones 1-4" for why this was broadened from single-recipient-only. Decision-outcome notifications (approve/reject results back to the employee) remain single-target. If `resolveTeamApprover` returns nothing at all (e.g. a department has no head assigned yet), the leave still auto-approves and `notifyOfficeAdmins` is the only notification sent (mirrored in `attendance.service.ts`'s `requestAdjustment`). There is still no admin-wide "see every notification" inbox ŌĆö `GET /notifications` is always scoped to the caller's own `employeeId`, `SUPER_ADMIN` included; `notifyOfficeAdmins` works by writing one row per admin, not by changing that scoping.

### Scheduled jobs

This app has no cron dependency or job queue ŌĆö the only scheduled work is the leave lifecycle sweep (`apps/api/src/services/leave-scheduler.service.ts`, `runLeaveLifecycleSweep`), piggybacked onto a second `setInterval` in `index.ts` alongside the existing DB keep-alive (runs once at boot, then every 15 minutes): auto-rejects any `PENDING` leave application once its `startDate` arrives (except Sick Leave ŌĆö see below), and sends a one-time `LEAVE_REMINDER` to the resolved approver the day before (guarded by `LeaveApplication.reminderSentAt`). Both operations are idempotent/re-run-safe by design (atomic `updateMany` guards), since nothing about this interval is transactional or coordinated across multiple API instances. If this app ever needs a *real* scheduler (precise time-of-day firing, distributed locking across instances, retries), replace this rather than piling more logic onto the interval.

### File storage

Supabase Storage replaces S3/R2. Buckets: `avatars` (public), `documents` (private), `payslips` (private). Never stream files through the API ŌĆö generate presigned upload/read URLs via `createSignedUploadUrl`/`createSignedReadUrl` in `apps/api/src/services/storage.service.ts`, which wraps the service role client (`apps/api/src/config/supabase.ts`).

### Tax engine

`packages/utils/src/tax.ts` exports `calculateIncomeTax(annualGross, officeCode, year)`. Routes to `calculateBDTax` or `calculateUKPAYE`. Never inline tax logic in the payroll service ŌĆö always call this function. Tax slabs are updated here annually.

### Prisma schema notes

- All IDs use `cuid()`.
- `Office.code` is the stable string key ("BD" / "UK") ŌĆö prefer it over `officeId` in business logic.
- `LeaveType.approvalChain` is a `Json` column storing `[{level: number, role: string}]` ŌĆö **no longer read for routing** (see RBAC's reporting-chain paragraph above); kept in schema only for historical data, superseded by `approver-resolution.service.ts`. `LeaveApprovalHistory.level` is likewise a display-only historical field now, since approval is single-step-with-override rather than sequential.
- `Notification` table must have Realtime enabled in Supabase for push to work.
- `Session` rows back refresh tokens 1:1 (see Auth flow); `User.isTwoFactorEnabled`/`twoFactorSecret` gate the 2FA step.
- Schema requires both `DATABASE_URL` (pooled, Supabase) and `DIRECT_URL` (direct, for migrations).
